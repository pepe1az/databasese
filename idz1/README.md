UNF: ненормализованная таблица

На первом этапе создана таблица orders_raw, имитирующая Excel-выгрузку.

Особенность таблицы: в одной строке одновременно хранятся данные заказа, клиента, адреса и список товаров.

Проблемные поля:

product_names
product_prices
product_quantities

Они содержат несколько значений внутри одной ячейки.

Пример:
```
product_names      = 'Смартфон, Чехол'
product_prices     = '60000, 1200'
product_quantities = '1, 2'
```
Это нарушает атомарность данных.

Аномалии в orders_raw

Подробные примеры приведены в файле:

checks/anomalies.txt

Кратко:

Аномалия вставки

Нельзя добавить новый товар без создания заказа, потому что отдельной таблицы товаров ещё нет.

Аномалия обновления

Данные клиента повторяются в разных заказах. Если клиент меняет телефон, нужно обновлять все строки с этим клиентом.

Аномалия удаления

Если удалить все заказы клиента, будут потеряны и данные самого клиента.

1NF

На этапе 1NF составные поля были разбиты на отдельные строки.

Было:

order_id = 1
product_names = 'Смартфон, Чехол'

Стало:

order_id = 1, product_name = 'Смартфон'
order_id = 1, product_name = 'Чехол'

Создана таблица:

orders_1nf

Количество строк увеличилось:

orders_raw = 1000 строк
orders_1nf = 2400 строк


2NF

На этапе 2NF были выделены отдельные сущности:

customers_2nf
products_2nf
orders_2nf
order_items_2nf

Частичные зависимости устранены следующим образом:

customer_name, customer_email, customer_phone зависят от клиента, а не от строки заказа
product_name, product_price зависят от товара, а не от строки заказа
quantity и price_at_order относятся к конкретному товару в конкретном заказе

Теперь данные клиента и товара не дублируются в каждой строке заказа.

На маленьких таблицах PostgreSQL часто выбирает Seq Scan, даже если индекс существует. Это нормально: для 200 строк полный проход дешевле чтения индекса.

Для демонстрации использования индекса применялось:

SET enable_seqscan = off;
pg_trgm

Обычный B-tree индекс не помогает для поиска вида:

WHERE name ILIKE '%42%'

Причина: шаблон начинается с %, поэтому поиск не может эффективно использовать обычный индекс.

Для такого поиска подключено расширение:

CREATE EXTENSION IF NOT EXISTS pg_trgm;

И создан GIN-индекс:

CREATE INDEX idx_customers_name_trgm
ON customers
USING gin (name gin_trgm_ops);

В checks/trgm_demo.txt видно, что индекс может использоваться через:

Bitmap Index Scan on idx_customers_name_trgm
Денормализация
Материализованное представление

Создано materialized view:

mv_monthly_sales

Оно заранее хранит агрегированные продажи по месяцам, товарам и категориям.

Сравнение из checks/mv_vs_join.txt:

JOIN + GROUP BY:
Execution Time: 3.323 ms

Materialized View:
Execution Time: 0.050 ms

Materialized view быстрее, потому что агрегаты уже посчитаны заранее.

Цена такого подхода:

данные могут устаревать
нужно выполнять REFRESH MATERIALIZED VIEW
занимает дополнительное место
Избыточное поле в таблице

В таблицу orders добавлено поле:

customer_name

Это копия customers.name.

Плюсы:

можно получить имя клиента без JOIN
можно сохранить исторический снимок имени клиента на момент заказа

Минусы:

нужна синхронизация через код приложения или триггер
OLTP vs OLAP
Характеристика	OLTP (PostgreSQL)	OLAP (ClickHouse)
Модель хранения	Строковая	Колоночная
Типичный запрос	Получить или изменить один заказ, клиента, товар	Посчитать агрегаты по миллионам строк
Нормализация	Обычно нормализованная схема	Часто денормализованная схема
Транзакции	Есть ACID-транзакции	Ограниченная поддержка транзакционности
Вставка	Частые небольшие INSERT/UPDATE	Массовые batch-вставки
Обновление/удаление	Эффективные точечные UPDATE/DELETE	UPDATE/DELETE дорогие и используются реже
Масштабирование	Вертикальное, репликация, шардинг сложнее	Хорошо масштабируется горизонтально
Типичный use case	Интернет-магазин, банковские операции, CRM	Аналитика, логи, метрики, отчёты
Вывод

Нормализация уменьшает дублирование данных и защищает от аномалий вставки, обновления и удаления.
Для OLTP-сценариев нормализованная схема удобна и безопасна.

Денормализация может ускорить чтение и отчёты, но добавляет избыточность, риск рассинхронизации и необходимость обновлять производные данные.
