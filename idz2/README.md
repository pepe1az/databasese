# ИДЗ-2 — ClickHouse: OLAP, MergeTree, денормализация

## Цель работы

Развернуть ClickHouse, спроектировать OLAP-схему под аналитическую нагрузку, загрузить 1M+ строк интернет-магазина и сравнить подходы PostgreSQL (OLTP, 3NF) и ClickHouse (OLAP, flat schema).

---

# Часть 1. Архитектурный подход

## Почему в ClickHouse используется денормализация

В отличие от PostgreSQL, где данные обычно приводятся к 3NF для минимизации избыточности и безопасных транзакционных обновлений, ClickHouse ориентирован на аналитическое чтение больших объёмов данных.

### Причины:

### 1. JOIN в OLAP дорогой

* ClickHouse оптимизирован под массовое сканирование колонок.
* JOIN нескольких крупных таблиц значительно замедляет запросы.
* Поэтому выгоднее хранить готовые плоские записи.

### 2. Избыточность компенсируется сжатием

* Повторяющиеся значения (`category`, `region`, `status`) хорошо сжимаются.
* Колоночное хранение уменьшает стоимость дублирования.

### 3. `LowCardinality` заменяет справочники

* Вместо отдельных таблиц категорий/регионов используются словарные кодировки.
* Ускоряет фильтрацию.
* Уменьшает размер хранения.

# Часть 2. Таблицы

## `orders_flat` — основное хранилище

* Движок: `MergeTree`
* 1 строка = 1 позиция заказа
* `ORDER BY (category, toStartOfHour(order_datetime), order_status)`

### Почему такой ORDER BY:

* улучшает сжатие;
* ускоряет аналитические выборки по категориям;
* ускоряет диапазонные запросы по времени;
* группирует похожие значения рядом.

---

## `orders_ttl`

* Используется для хранения данных с автоматическим удалением старых записей.
* TTL: `order_date + INTERVAL 90 DAY DELETE`

### Назначение:

* автоматическое управление retention policy;
* уменьшение объёма горячих данных;
* демонстрация lifecycle management.

---

## `monthly_sales`

* Движок: `SummingMergeTree`
* Предвычисленные агрегаты:

  * выручка;
  * количество товаров;
  * число заказов.

### Преимущества:

* мгновенные агрегатные запросы;
* снижение нагрузки;
* типичный OLAP-подход.

### Недостаток:

* snapshot-таблица требует обновления;
* в production лучше использовать Materialized View.

# Часть 5. TTL

### Демонстрация:

* Старые записи вставлены;
* TTL применён;
* После `OPTIMIZE FINAL` устаревшие строки удаляются автоматически.

### Вывод:

ClickHouse поддерживает автоматическое lifecycle-управление данными без ручного удаления.

# Часть 6. Сжатие

## Общая статистика:

* Compressed: **45.50 MiB**
* Uncompressed: **107.62 MiB**
* Compression ratio: **2.37x**

## Лучше всего сжимаются:

* `category`
* `region`
* `order_status`
* `customer_email`

### Почему:

* Низкая кардинальность;
* Повторяемость;
* Dictionary encoding (`LowCardinality`).

# Часть 7. PostgreSQL vs ClickHouse

| Запрос / Операция         | PostgreSQL (3NF) | ClickHouse (flat)           | Вывод                      |
| ------------------------- | ---------------- | --------------------------- | -------------------------- |
| Вставка 1 строки          | Быстро           | Медленнее для single insert | PG лучше для OLTP          |
| Массовая вставка          | Хорошо           | Отлично                     | CH эффективен batch insert |
| Топ-10 товаров (1M строк) | Медленнее        | ~0.059 сек                  | CH значительно быстрее     |
| JOIN 4 таблиц             | Требуется        | Не нужен                    | Flat schema быстрее        |
| UPDATE статуса            | Полноценный      | Ограниченный                | PG лучше                   |
| TTL / retention           | Ручной           | Нативный TTL                | CH удобнее                 |
| Размер на диске           | Выше             | Ниже                        | CH эффективнее             |
| Подстрочный поиск         | Индексы нужны    | Работает быстро             | CH хорош для аналитики     |

# Общий вывод

## PostgreSQL (OLTP)

Подходит для:

* транзакций;
* обновлений;
* строгой нормализации;
* бизнес-логики.

## ClickHouse (OLAP)

Подходит для:

* аналитики;
* агрегаций;
* дешёвого хранения больших объёмов;
* BI/отчётности;
* time-series аналитики.

---

# Финальный вывод

Практическое сравнение показало:

* PostgreSQL лучше для операционной деятельности;
* ClickHouse радикально быстрее на аналитических запросах;
* Денормализация в OLAP оправдана;
* `LowCardinality`, `MergeTree`, `SummingMergeTree`, TTL и колоночное хранение дают серьёзный прирост производительности.

**Главное:**

> OLTP хранит данные для работы системы.
> OLAP хранит данные для анализа системы.
